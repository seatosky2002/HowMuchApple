# 번개장터 검색 API 동작 정리 & 서울·경기 전량 크롤 결과

실험 일자: 2026-07-17. Oracle 서버(`howmuch-backend` 컨테이너)와 로컬에서 실제 요청으로 검증했다.

## 검색 API 기본

```
GET https://api.bunjang.co.kr/api/1/find_v2.json
헤더: User-Agent: Mozilla/5.0, Referer: https://m.bunjang.co.kr/
```

| 파라미터 | 설명 |
|---|---|
| `q` | 검색어 |
| `order` | 정렬 (아래 표) |
| `page` / `n` | 페이지 번호 / 페이지 크기 |
| 응답 | `num_found`(총 건수), `list`(매물 배열) |

## 핵심 발견 ①: 기존 코드의 `lat`/`lon`/`distance`는 아무 효과가 없다

기존 `bunjang.py`는 `lat=37.5665, lon=126.9780, distance=0`을 항상 붙였지만,
**좌표를 천호동/수원으로 바꾸고 distance를 0/3/3000으로 바꿔도 결과가 완전히 동일**했다
(num_found·매물 목록 모두 같음). 즉 위치 필터링은 처음부터 동작한 적이 없고 실제로는 전국 검색이었다.

## 핵심 발견 ②: `order=distance`는 좌표 기준 "거리순 정렬"이다 (필터 아님)

`order=distance` + `lat`/`lon`을 주면 num_found는 그대로지만 **가까운 매물부터 정렬**된다.
서울시청(37.5665, 126.9780) 기준 "아이폰 14 프로" 검색(n=100) 실측:

| 페이지 | 내용 |
|---|---|
| 0 | 서울 100건 (도심부터) |
| 3 | 서울 94 + 경기 6 (경계 진입) |
| 10 | 전부 지방 (충북·대전) |
| 20~ | 전부 위치 없음 (맨 뒤로 밀림) |
| 60 | 빈 페이지 (num_found=4006 소진) |

→ **한 중심점에서 페이지를 넘기다 서울/경기 매물이 안 나오면 중단**하는 것만으로
지역 내 위치 등록 매물을 전량 수집할 수 있다. 동(행정구역)별 반복 검색이 필요 없다.

## 정렬(`order`) 유효값

| 값 | 동작 |
|---|---|
| `date` | 최신순 (기본) |
| `score` | 정확도순 — num_found가 더 크게 나옴 (매칭이 느슨해짐: 4006 → 4460) |
| `distance` | `lat`/`lon` 기준 거리순. 위치 없는 매물은 맨 뒤 |
| `price_asc` / `price_desc` | 가격 오름/내림차순 |
| `popular` | 인기순 |

## 페이지네이션 한계

- `n`(페이지 크기) 최대 **200** (`n=300`은 HTTP 400)
- 접근 가능한 오프셋(page×n)은 **10,000 미만** (offset 9,900은 OK, 10,000은 400)
  - 즉 한 검색어로 최대 10,000건까지만 볼 수 있다. "아이폰"처럼 num_found가 61,177인
    검색어는 전체 열람이 불가능하므로, 필터나 정렬로 잘라서 접근해야 한다.
  - 모델 단위 검색어(num_found 수천 건)는 한계 안에 넉넉히 들어온다.

## 동작하는 필터 파라미터 (실측)

"아이폰 14 프로" 기준 num_found 4,006에서의 변화:

| 파라미터 | 효과 | 실측 |
|---|---|---|
| `f_price_min` | 최소 가격 | 500000 → 1,560건 |
| `f_price_max` | 최대 가격 | 500000 → 2,522건 |
| `f_category_id` | 카테고리 한정 | 600700001 → 1,864건 |
| `f_used=2` | 새상품(미개봉류)만 | 1,054건 |

무효(무시됨): `price_min/max`, `category_id`, `used`, `f_bizseller`, `free_shipping`,
그리고 위의 `lat`/`lon`/`distance`.

## 매물 필드 관찰

- `location`: 검색 결과(최신순)에서 **약 25%만 보유**. 직거래 위치를 등록한 매물만 있다.
- `status`: 검색 결과는 전부 `"0"` — 판매중만 노출되는 것으로 보임.
- `only_neighborhood`: 표본 100건에서 0건.
- `ad`: 이번 크롤 결과 7,366건 중 광고 0건 (거리순 정렬에는 광고가 섞이지 않는 듯).
- 상세 API `GET /api/pms/v1/products/{pid}/detail/web`의 `data.product.geo`에
  **정확한 좌표(lat/lon)와 행정동 주소**가 있다 (예: `서울특별시 종로구 종로5.6가동`).

## 키워드 토큰화

공백 변형은 결과가 거의 동일하다: `아이폰 14 프로` 4,006 / `아이폰14프로` 4,143 /
`iPhone 14 Pro` 4,006 / `iphone14pro` 3,993. 별칭 키워드의 추가 효과는 크지 않다
(크롤에서는 pid로 전역 중복 제거).

## 서울·경기 전량 크롤 방법론 & 결과 (2026-07-17)

- 실행: Oracle 서버 `howmuch-backend` 컨테이너, `PYTHONPATH=/app`으로
  앱의 `CRAWL_TARGETS`(72개 타겟, 키워드 227개)와 `matches_target_listing` 필터 재사용
- 방식: 타겟×키워드마다 서울시청 중심 `order=distance`, `n=100`으로 페이지 순회.
  서울/경기 매물 0건 페이지가 2연속이면 해당 키워드 중단. pid 전역 중복 제거.
- 요청 총 1,188건, 간격 0.35s, **429 차단 없음**

## 검증: 동네별 검색 vs 단일 중심점 검색은 결과가 동일하다

"아이폰 14 프로"를 서로 다른 중심점 3곳(서울시청 / 수원 인계동 / 의정부 의정부동)에서
각각 거리순 전체 순회(41페이지)한 뒤, 수집된 서울/경기 매물 pid 집합을 비교했다:

- 세 중심점 모두 **정확히 같은 1,001건** (집합 차이 0건)
- 기준 변동폭(같은 중심점 2회 순회 차이)도 0건 — 시간차 노이즈 아님

`order=distance`는 동일한 전국 결과 집합을 정렬만 바꿔 보여주는 것이므로,
어느 좌표를 중심으로 잡아도 서울/경기 매물의 집합은 불변이다.
동네별 반복 검색은 추가 매물 없이 요청 수만 동네 수만큼 배가된다.

## `bunjang.py` 개선 반영 (2026-07-17 배포 완료)

`app/crawlers/bunjang.py`를 위 방법론으로 재작성해 로컬 리포와 Oracle 서버
(이미지 리빌드 + 재시작) 양쪽에 반영했다:

1. 무효 파라미터(`distance=0` 필터 흉내) 제거 → `order=distance` + 서울/경기 소진 시 중단
2. `n=100` → `n=200` (요청 수 절반)
3. 모델당 30건 상한(`default_items_per_target`) 미사용 — `max_items`만 존중 (당근 크롤러와 동일 컨벤션)
4. 위치 없는 매물은 수집 대상에서 제외되므로 상세 페이지 지역 조회(`_fetch_detail_regions`) 삭제
5. 429 시 30s×시도 횟수 백오프

배포 검증: 컨테이너 내 스모크 테스트에서 iPhone 16e 단독 77건(서울 49/경기 28) —
로컬 테스트와 동일 수치.
